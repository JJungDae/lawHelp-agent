# 공통 작업지시

## 1. 문서 목적

이 문서는 팀원이 같은 기준으로 개발하고, Codex/Claude 같은 로컬 코딩 에이전트가 프로젝트 범위를 과도하게 확장하지 않도록 하기 위한 공통 작업 기준이다.

구체적인 개별 프롬프트는 역할 분배 이후 각자 작성하되, 모든 작업은 이 문서의 원칙을 우선한다.

---

## 2. 최우선 구현 원칙

이번 프로젝트의 핵심은 기능을 많이 넣는 것이 아니라, 아래 흐름을 실제로 끝까지 동작시키는 것이다.

```text
사용자 질문
→ 생활법률 범위 판단
→ ChromaDB 검색
→ Upstage Solar 기반 답변 생성
→ LiteLLM wrapper 기반 Guardrail 적용
→ SSE로 Streamlit 화면 출력
→ Docker Compose 실행
→ GCE 배포
→ GitHub Actions에서 Ruff/pytest 통과
```

개발 원칙은 다음과 같다.

```text
얇게 끝까지 먼저 관통시킨다.
Mock으로 먼저 성공시킨다.
그 다음 실제 LLM, ChromaDB, SSE, 배포를 붙인다.
```

---

## 3. 구현 우선순위

### P0: 반드시 구현

- 백문백답 샘플 데이터 JSON/CSV 정리
- ChromaDB 적재 스크립트
- 질문 → ChromaDB top-k 검색
- 검색 결과 기반 Upstage Solar 답변 생성
- LiteLLM wrapper 기반 LLM 호출 통일
- Guardrail 적용
- FastAPI `/health`, `/chat/sync`, `/chat/stream`
- SSE 스트리밍
- Streamlit 최소 UI
- Docker Compose 실행
- GCE VM 배포
- GitHub Actions + Ruff + pytest

### P1: P0 완성 후 구현

- Retry/Fallback
- LLM timeout, rate limit, API 오류 발생 시 1회 재시도
- 재시도 실패 시 fallback 응답 반환
- 검색 결과 부족 시 “질문을 더 구체적으로 입력해 주세요” 안내

### P2: 시간이 남으면 구현

- 명확화 되묻기
- 모든 질문에 적용하지 않는다.
- 대표 시나리오 1개, 예: 기초생활보장 질문에서만 구현한다.

---

## 4. 이번 프로젝트에서 하지 않는 것

아래 기능은 MVP에서 제외한다.

- 답변별 출처 인용 UI 표시
- 판례 전문 검색
- 소장, 고소장, 계약서 등 법률문서 작성
- 개별 사건의 승소 가능성 판단
- 개인정보 수집 및 저장
- 로그인/회원관리
- 장기 기억
- 대화 이력 DB 저장
- Supabase 연동
- LiteLLM Proxy/Admin UI 기반 운영

단, 데이터 출처는 README와 발표자료에 명시한다. 검색 문서 내부에는 `id`, `category`, `question` 정도의 메타데이터를 유지할 수 있다.

---

## 5. LiteLLM 구현 기준

이번 프로젝트에서는 LiteLLM을 **wrapper 우선 방식**으로 구현한다.

즉, 별도의 LiteLLM Proxy/Admin UI 컨테이너를 처음부터 띄우지 않는다. FastAPI와 LangGraph의 각 노드는 Upstage API를 직접 호출하지 않고, `app/core/llm.py`의 LiteLLM wrapper를 통해 호출한다.

### LiteLLM wrapper의 목표

- LLM 호출부를 한 곳으로 통일한다.
- 모델 호출 방식 변경 시 수정 범위를 줄인다.
- Guardrail, Retry, Fallback을 공통 계층에서 관리할 수 있게 한다.
- LLM 관련 예외를 공통 예외로 변환한다.

### 예상 구조

```text
app/core/llm.py
  ├─ generate_text()
  ├─ stream_text()
  ├─ classify_text()
  ├─ apply_guardrail()
  └─ call_with_retry()
```

---

## 6. Guardrail 구현 기준

Guardrail은 P0 핵심 기능이다.

### 차단 대상

- 승소 가능성 판단 요청
- 소장, 고소장, 계약서 등 법률문서 작성 요청
- 구체적 법률 자문 요청
- 주민등록번호, 계좌번호 등 개인정보 입력 또는 저장 요청
- 프롬프트 인젝션
- 생활법률 백문백답 범위 밖 질문

### 처리 방식

입력 단계에서 위험 요청을 감지하면 RAG 검색과 답변 생성을 진행하지 않고 즉시 안내 메시지를 반환한다.

출력 단계에서는 답변이 법률 자문처럼 단정적으로 표현되지 않았는지 검사한다.

차단 메시지는 다음 원칙을 따른다.

```text
이 서비스는 법제처 생활법령 백문백답을 바탕으로 일반 정보를 안내하는 서비스입니다.
개별 사건의 승소 가능성 판단, 법률문서 작성, 구체적 법률 자문은 제공하지 않습니다.
필요한 경우 대한법률구조공단 등 전문기관 상담을 이용해 주세요.
```

## 7. 출처 표시에 대한 개발 기준

최종 기획서 일부 시나리오에는 “출처와 함께”라는 표현이 남아 있을 수 있다.

하지만 실제 MVP에서는 답변마다 출처 링크를 UI에 표시하지 않는다.

개발 기준은 다음과 같다.

- 사용자 화면에는 답변 본문과 일반 정보 제공 고지를 우선 표시한다.
- 데이터 출처는 README와 발표자료에 명시한다.
- 내부 검색 문서에는 `id`, `category`, `question` 메타데이터를 유지한다.
- 시간이 남을 경우에만 검색된 백문백답 제목 정도를 보조 정보로 보여줄 수 있다.

---

## 8. 근거 충실성 기준

LLM이 검색 결과에 없는 내용을 단정적으로 생성하지 않도록 한다.

완전한 무오류를 보장한다는 의미가 아니라, 테스트 질문 기준으로 다음을 점검한다.

- 검색 결과에 없는 법률 내용을 단정하지 않는가?
- “~해야 합니다”처럼 법적 지시로 보이는 표현을 남발하지 않는가?
- 불확실한 경우 “확인 필요”, “전문기관 상담 권장”으로 처리하는가?
- 답변 말미에 “일반 정보 제공이며 법률 자문이 아님”을 포함하는가?

# Git 충돌 방지 원칙

## 10. 공통 Git 원칙

- `main` 브랜치에 직접 작업하지 않는다.
- 기능별 branch를 만든다.
- 작업 완료 후 Pull Request 또는 팀 합의 후 merge한다.
- 같은 파일을 동시에 수정하지 않는다.
- 매일 작업 시작 전 `main`을 최신화한다.
- Claude/Codex가 만든 코드라도 설명할 수 없으면 merge하지 않는다.

## 12. 커밋 메시지 규칙

```text

docs: 문서 수정
refactor: 구조 개선

```

예시:

```bash
git commit -m "feat: add chat sync endpoint"
git commit -m "fix: handle empty search result"
git commit -m "docs: add implementation guide"
```

## 13. 충돌을 줄이는 파일 담당 기준

| 영역 | 주 작업 단위 | 비고 |
|---|---|---|
| `data/` | 샘플 데이터 관리 | JSON/CSV |
| `scripts/ingest_chroma.py` | ChromaDB 적재 | 데이터 적재 담당 |
| `app/repositories/` | ChromaDB 접근 | 검색 함수 |
| `app/api/` | FastAPI endpoint | `/chat/sync`, `/chat/stream` |
| `app/agents/` | LangGraph 흐름 | scope/retrieve/generate/guardrail |
| `app/core/llm.py` | LiteLLM wrapper | LLM 호출 공통 계층 |
| `frontend/` | Streamlit UI | 화면 담당 |
| `tests/` | 테스트 | pytest |
| `docs/` | 개발 문서 | 공통 기준 |

## 14. 충돌 주의 파일

아래 파일은 작업 전에 팀원에게 공유하고 수정한다.

```text
README.md
.env.example
docker-compose.yml
app/schemas/
app/core/config.py
```

이 파일들은 프로젝트 전체 계약에 해당하므로 혼자 큰 폭으로 바꾸지 않는다.

---

## 15. Merge 전 설명 기준

작업을 합치기 전, 각자 아래 네 가지를 설명한다.

```text
1. 어떤 파일을 수정했는가?
2. 이 코드는 어떤 입력을 받는가?
3. 이 코드는 어떤 출력을 만드는가?
4. 다른 파일과 어디에서 연결되는가?
```

이 네 가지를 설명하지 못하면 merge하지 않는다.

---

## 16. 로컬 코딩 에이전트 공통 작업 원칙

로컬 코딩 에이전트에게 작업을 맡길 때는 다음 기준을 반드시 포함한다.

```text
- 최종 기획서와 공통_작업지시.md를 우선 기준으로 삼는다.
- MVP 범위 밖 기능은 구현하지 않는다.
- Supabase, 로그인, 장기 기억, 출처 표시 UI는 구현하지 않는다.
- ChromaDB 기반 RAG 흐름을 우선한다.
- LiteLLM은 wrapper 우선 방식으로 구현한다.
- 전체 구조를 바꾸지 말고 필요한 파일만 최소 수정한다.
- 코드를 작성하기 전에 어떤 파일을 왜 수정할지 먼저 설명한다.
- 구현 후 실행 명령어와 테스트 방법을 알려준다.
- 데이터가 어디에서 들어와 어디로 나가는지 설명한다.
```
